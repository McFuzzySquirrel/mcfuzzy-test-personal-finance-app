# Decisions Made
- Decision: Use `schema_version` table for migration version tracking while retaining compatibility with existing migration IDs.
  - Reason: Explicit numeric version tracking is required by the database-engineer contract.
  - Impact: Future migrations can be added and skipped deterministically.
- Decision: Implement query helpers in dedicated `db/*.ts` modules and expose via `db/index.ts`.
  - Reason: Keep app code free of raw SQL and align with mode responsibilities.
  - Impact: Feature hooks/screens can consume stable typed APIs.
- Decision: Seed default categories as part of `initializeDatabase()` after migrations.
  - Reason: Startup should be app-ready without separate manual seed calls.
  - Impact: Default categories are guaranteed for first launch and tests can invoke same initialization path.
- Decision: Route all expense and category screen data access through `useExpenses` and `useCategories`.
  - Reason: The handoff explicitly requires DB hooks only and screens need a narrow API surface.
  - Impact: UI code stays decoupled from SQL modules while still supporting refresh on focus and mutation.
- Decision: Use the Dashboard tab as the Add Expense entry point and transaction rows as the Edit Expense entry point.
  - Reason: Stack routes were already wired; the missing work was making those routes reachable in the actual UI.
  - Impact: The new screens are discoverable without restructuring navigation.

# Key Learnings
Technical insights
- Existing foundation used `_migrations`; contract now requires `schema_version`, so migration runner must be updated.

Prompting insights
- The request explicitly requires startup seed call path, not only a seed helper function.

Tooling insights
- Environment includes `python3` but not `python`.
- Jest + Expo runtime in this environment may require additional native-compatible dependencies and Node version alignment before DB integration tests can execute.
- React Native lint rules in this repo are strict enough that the new screen/component styles need to avoid unnecessary literals and remain minimal.
