# Decisions Made
- Decision: Move Jest preset from `jest-expo` to `react-native` for current QA baseline.
  - Reason: `jest-expo` bootstrapping invoked Expo Winter runtime paths that failed in this environment with import-scope errors before any test code executed.
  - Impact: Tests execute reliably without production behavior changes; Expo-specific modules are mocked in setup where needed.
- Decision: Keep interim coverage thresholds low but explicitly documented in config.
  - Reason: During bootstrap stabilization the immediate blocker is test execution; strict gates should be restored after broader suite coverage is implemented.
  - Impact: CI/local tests are unblocked now while preserving a clear follow-up target in config comments.
- Decision: Keep split/lent/borrowed fields as controlled UI state in `add-expense` and avoid DB calls inside sub-forms.
  - Reason: The agent contract requires `SplitExpenseForm` and `LentBorrowedForm` to be controlled/presentational while the parent persists records.
  - Impact: Add-expense preserves a fast default path while still supporting advanced debt/split flows without coupling form components to persistence.
- Decision: Exclude `lent`/`borrowed` in SQL aggregations used by dashboard and budget summaries.
  - Reason: PRD constraints specify debt tracking entries must not contribute to budget spend totals or spent-this-month calculations.
  - Impact: Dashboard/budget totals reflect actual spending categories while debt records remain visible in transactions/splits only.
- Decision: Use a dynamic-import wrapper component for the Insights route.
  - Reason: It still defers chart-screen module evaluation, but unlike a `require()`-based `getComponent` approach it complies with the repo lint rules.
  - Impact: The dashboard and add-expense path remain free from eager chart-screen loading while validation stays clean.
- Decision: Keep `WeeklySummary` independent from `react-native-gifted-charts`.
  - Reason: The dashboard contract explicitly requires a lightweight self-contained widget and warns against heavy chart imports there.
  - Impact: The dashboard gets the weekly summary feature without taking on the Insights chart dependency path.
- Decision: Build insights hooks on top of existing expense aggregation helpers.
  - Reason: The current DB layer already exposes the required monthly and weekly totals, so new SQL would be redundant.
  - Impact: Phase 3 stays focused on composition, rendering, and accessibility rather than expanding the data layer.
- Decision: Keep export formatting functions pure and move native operations (print/share/file write) behind dedicated wrappers.
  - Reason: The export contract requires side-effect boundaries that remain unit-testable even when native modules are missing.
  - Impact: `generateCsv` and `generatePdfHtml` are deterministic pure functions while device-specific capabilities fail gracefully with explicit user feedback.
- Decision: Mount export entry from Transactions using existing month, category, expense, and budget hooks.
  - Reason: This avoids duplicating retrieval logic and keeps export scope aligned with the user-selected month context.
  - Impact: Export output mirrors current Transactions month filters and uses existing app data flow.
- Decision: Keep user confirmation as a mandatory pre-share gate in modal flow.
  - Reason: PRD SP-04 requires explicit acknowledgement before financial data can be shared off-device.
  - Impact: Share sheet is never opened without explicit user confirmation.
- Decision: Raise coverage gates to pragmatic non-trivial thresholds during continuation.
  - Reason: `1%` thresholds no longer reflected project maturity after multiple implemented phases and passing suites.
  - Impact: Future regressions are less likely to silently pass CI while still allowing incremental test growth.
- Decision: Reject negative currency inputs at parser entry.
  - Reason: Input sanitizer stripped `-` and could convert `-10` into a valid positive value, violating expected validation behavior.
  - Impact: Add/edit amount parsing now aligns with form validation intent and tests.
- Decision: Prefer JDK 17 in local Detox build bootstrap when available.
  - Reason: Android/Gradle compatibility is more reliable on JDK 17 across RN/Expo toolchains; user environment had Java installed but not pinned for build.
  - Impact: E2E scripts now auto-select a compatible Java home where present and provide deterministic behavior in CI and local runs.
- Decision: Extend smoke E2E beyond route checks into one data-creating happy path.
  - Reason: A creation-and-verification flow catches integration regressions that pure navigation smoke tests miss.
  - Impact: Detox baseline now validates one core product action (adding an expense) end-to-end.
- Decision: Add test coverage for app bootstrap persistence behavior before further feature expansion.
  - Reason: `app_state` and default seeding are foundational startup behaviors that can regress silently without direct tests.
  - Impact: DB bootstrap behavior is now covered by unit tests, improving confidence in first-launch and month-rollover flows.
- Decision: Standardize on Java 21 for local and CI Android/Detox tooling.
  - Reason: User preference and security posture favor Java 21, and testing showed the actual incompatibility was Gradle 9 rather than Java 21 itself.
  - Impact: E2E scripts and CI now target Java 21, while the build script patches the generated wrapper to Gradle 8.13 for compatibility.
- Decision: Use a single Android emulator Detox configuration (`android.emu.debug`) with one smoke spec as baseline.
  - Reason: User requested minimal deterministic setup replacing placeholders without broad production-code churn.
  - Impact: E2E path is now runnable and CI-integrated while remaining low-maintenance.
- Decision: Build APKs via Expo prebuild fallback plus Gradle in Detox app build command.
  - Reason: Repo currently does not track Android native project files, so Detox needs a repeatable way to generate/build artifacts.
  - Impact: CI can build from managed Expo source; local runs require Android + Java toolchain.
- Decision: Keep smoke assertions on existing testIDs instead of adding new UI selectors.
  - Reason: Existing IDs already cover launch/dashboard/add-expense navigation for a realistic baseline.
  - Impact: No production component changes were needed for baseline e2e coverage.
